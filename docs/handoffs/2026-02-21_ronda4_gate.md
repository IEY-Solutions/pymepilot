## 3 accionables de Ronda 4 — PENDIENTES de correccion

Estos NO fueron aplicados al plan. Quedaron pendientes cuando Pato pidio el GATE.

### Accionable #1: psql -U pymepilot_app necesita -h 127.0.0.1

- **Agente:** @db-architect
- **Ubicacion:** Test 12 Parte D y Paso 12.5 del plan
- **Problema:** Los comandos `docker exec ... psql -U pymepilot_app` fallan con
  peer authentication porque el container usa auth trust solo para conexiones TCP,
  no para socket Unix. Agregar `-h 127.0.0.1` lo resuelve.
- **Accion:** Actualizar los comandos psql en Test 12D y Paso 12.5 del plan.
- **Riesgo si no se corrige:** Pato no puede ejecutar los tests de aislamiento
  con pymepilot_app. Bloqueante para paso 12.5.

### Accionable #5: Sin delay entre fetch_customers/products/orders

- **Agente:** @python-engine
- **Ubicacion:** sync.py pasos 7-9 (lineas 707-709)
- **Problema:** time.sleep(SYNC_RATE_LIMIT_DELAY) existe ENTRE paginas de una entidad
  pero NO entre las 3 llamadas a fetch_*. Una rafaga de 30 requests totales
  (10 paginas x 3 entidades) solo tiene delay intra-entidad, no inter-entidad.
- **Accion:** Agregar time.sleep(SYNC_RATE_LIMIT_DELAY) entre pasos 7, 8, 9.
- **Riesgo si no se corrige:** Bajo. Contabilium podria responder 429 al inicio
  de la segunda entidad, pero el retry con backoff lo maneja.

### Accionable #7: conn.commit() explicito entre pasos 11 y 12

- **Agente:** @python-engine
- **Ubicacion:** sync.py pasos 11-12 (lineas 784-800)
- **Problema:** Pasos 11 (actualizar sync_log a completed) y 12 (audit de logs)
  comparten una transaccion implicita de psycopg3. Si paso 12 falla con
  excepcion, el UPDATE de paso 11 se pierde. El finally lo cubre
  (cambia a 'failed'), pero un sync exitoso apareceria como failed.
- **Accion:** Decidir una de dos opciones:
  A. Agregar conn.commit() entre pasos 11 y 12 (paso 11 se persiste independientemente)
  B. Documentar que el finally cubre este caso (aceptar el riesgo)
- **Riesgo si no se corrige:** Si audit_logs_for_secrets() explota (improbable),
  un sync exitoso queda como 'failed'. El proximo sync rehace todo (sin perdida).
